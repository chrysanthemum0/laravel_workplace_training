Git 

*깃을 이용하는 이유
- 팀단위 개발 및 버전관리에 매우 용이하기에 사용함
예시. 
1. 영업 그래프의 갱신
2. 주소신 데이터의 수정
3. 새로운 좌석표를 만듬 
4. 다음달 급여 정산 
이 있다고 하면 A씨와 B씨가 4의 업무를 맡게된다고 가정함. 

이 때, 최신 데이터를 가져올 필요가 있으므로 
그 최신 데이터를 remote(외부, 깃허브)에서 로컬(자신이 작업하는 워크스페이스)로가져옴
이것을 clone이라함 

이러한 과정에서 프로그램을 수정 완료 후 add로 스테이징 영역에 올린 후 
기록을 위해 commit을 진행함 

현재의 작업을 로컬에서 진행했기에 선배, 혹은 상급자에게 풀 리퀘스트를 받고 
remote에 올리는 과정

*중요한 것들 

1. 로컬 레포지토리의 작정 
- vs code에서 레포지토리 클론 
- 클론하고 싶은 레포지토리 ssh url 붙여넣기 
- 로컬에 저장할 경로 설정 ! c/users/git -> 연수 경로 
- 클론 한 리포지토리 열기 

2. 파일의 추가 
- 새로운 파일 생성  
- 소스 관리 들어가기 
- 변경파일 스테딩(add)에 올리기 
- 로컬 레포지토리에 반영(commit) -> commit메세지도 자세하게 남기기 이건 회사에서 선배하는 것 보고 배우기
- 리모트 레포지토리에 반영 push 하기

3. 파일의 변경 
- 변경할 파일 변경 후 위의 방법과 동일하게 진행

4. 파일의 삭제 
- 삭제할 파일 삭제 후 스테딩(add) -> commit -> push

5. 다른 사람이 갱신한 파일 받아오기 
- remote 레포지토리에서 갱신된 파일이 있다면 
- git graph 에서 소스 관리 우측 메뉴 petch 
- add commit push 등으로 파일 변경 및 삭제

리모트 feature/branch를 리모트 master에 병합시켜주고 싶은데, 리모트 master와 리모트 feature/branch의 내용이 달라서 머지가 안 되는 경우가 종종 생긴다. 
그 경우 conflict가 일어났다고 한다.

그냥 로컬과 리모트의 코드를 합치기 위해선 확인할 코드가 있다는 신호 정도로 받아들이면 된다.

Git 브랜치란
branch(브랜치)란 독립적으로 어떤 작업을 진행하기 위한 개념입니다. 
필요에 의해 만들어지는 각각의 브랜치는 다른 브랜치의 영향을 받지 않고 영향을 주지 않기 때문에 여러 작업을 동시에 실행할 수 있습니다. 
브랜치에서 작업한 부분은 merge(병합)라는 방법으로 기존에 있던 버전에 merge(병합)를 할 수 있다.
(여기서는 merge(병합)를 다루지는 않을 것입니다.)
